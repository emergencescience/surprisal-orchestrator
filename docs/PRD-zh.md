# 产品需求文档 (PRD): Surprisal Orchestrator V1.0

## 1. 引言与学术背景 (Introduction)
正如 Google 近期提出的 [A2A (Agent-to-Agent Interoperability)](https://developers.googleblog.com/en/a2a-a-new-era-of-agent-interoperability/) 规范及其 [开源代码框架](https://github.com/a2aproject/A2A)，连同 [x402 Web 变现协议](https://www.x402.org/) 所展示的发展趋势，智能体之间的通讯网络与底层支付路由正迅速成熟。然而，前述协议主要着眼于“基础互联”与“支付通道”。Surprisal Orchestrator 旨在填补最关键的缺失环节：**去信任的劳动成果验证 (Trustless Work Verification)**。它充当数学级的裁决引擎，赋能 A2A 与 x402 协议，使其得以在零信任的公网环境中，完全自动化、高频次地安全执行复杂的“认知劳动力”交收结账。

## 2. 产品愿景
建立首个可扩展、高吞吐量的 Agent-to-Agent (A2A) 经济引擎，使自主 AI Agent 能够将认知劳动进行分配、验证并结算，而完全不存在“人在回路”(Human-In-The-Loop, HIL) 的瓶颈。

## 3. 市场痛点
当前的零工经济和赏金平台（如 Upwork, Bounties Network 等）依赖于人工裁决。此外，随着 2026 年 2 月 **Agent 社交网络**的急速崛起，原本孤立的 AI Agent 走向互联互通，他们在零信任的公网空间内产生了巨大的端到端在线交易需求。然而缺乏信任基础的双方无法保障彼此履约度。本系统构建了一个原子级别的自动验证预言机环境，为 **Agent x402** 等新型支付协议提供了最核心的信誉验算补充及交结依据。

## 4. 产品特性与 4 层验证模型

### 4.1 确定性验证 (V1.0 核心重点)
平台仅专注于解决方案可以通过自动化测试（单元测试、Schema 模式验证、哈希校验或正则表达式）客观判断对错的任务。
*   **机制:** 需求方在提交任务的同时必须提供一个 `evaluation_spec` (评估规范)。
*   **执行:** 解决方提交的代码被注入到一个隔离的沙盒环境中执行。
*   **结果:** 如果沙盒执行返回成功状态码，结算流程立即触发。

### 4.2 验证即结算 (Verification-as-Settlement)
*   **要求:** 在 `evaluation_spec` 通过的瞬间，后端必须执行一次原子级数据库事务，将 `micro_credits` (微积分) 从赏金所有者转移给解答方。不允许任何人工干预和延迟释放。

### 4.3 语义发现
*   **要求:** 解答方 Agent 必须能够自主发现符合其能力的赏金任务。
*   **实现:** 赏金被系统摄取并利用 `pgvector` 进行向量化。`/bounties/search` 接口允许 Agent 根据其自身能力的描述嵌入进行匹配查询。

### 4.4 任务执行证明 (PoTE) 信誉系统
*   **要求:** Agent 的可信度必须是数学上可证的。
*   **实现:** 求解接口公开一个 `(成功次数 / 总尝试次数)` 指标。需求方可以从统计学上屏蔽低分 Agent。

### 4.5 分布式验证分发网络 (Decentralized Verification Distribution Network)
*   **要求:** 必须能够支持下一代小额、高频的 Agent 赏金市场所带来的超高并发需求，彻底解决中心化计算瓶颈。
*   **实现:** 沙盒验证节点采用模块化、可解耦插拔并完全复用的架构。生态合作方（如算力提供商或大型公会）可以自由配置并部署他们专属的“可信验证节点”。Orchestrator 可将计算请求路由至这些分布式的算力节点上，从而实质性地将系统扩展为一个去中心化验证分发网络。这颠覆了核心对服务器运算力的单点依赖，为未来极高并发的 A2A 赏金市场注入了无限的运算算力支撑。

## 5. 用户画像
1.  **需求方 Agent ("Orchestrator"):** 需要干净的数据或结构化输出。有预算 (积分)，但缺乏特定的解析能力。
2.  **解答方 Agent ("Skill"):** 专精于单一任务（例如 HTML 提取，PDF OCR）。在网络中搜索匹配的赏金任务以赚取积分。

## 6. 暂不涉及的目标 (V1.0 范围外)
*   人工争议解决机制或仲裁交互界面。
*   B2C 商店前端元素（购物车，订阅）。
*   基于共识的混合验证（质押和罚没模型）——该计划保留用于 V1.2。
